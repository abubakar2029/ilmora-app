[2026-06-02] Chat: sent message shows one tick (✓) then disappears
--------------------------------------------------------------------------------
Symptom:
  User sends a message; bubble appears briefly with a single check mark, then
  vanishes. Thread list may still look empty or unchanged.

Root cause:
  useConversation polls the server every 10s. While the message was still
  optimistic (negative temp id), a background GET could return the old message
  list and React Query replaced the whole cache — wiping the optimistic bubble.
  (Same class of bug as an earlier fix that removed invalidateQueries on send;
  polling refetch reintroduced the wipe.)

Fix:
  - src/lib/chat-cache.ts — mergeConversationFromServer() keeps optimistic and
    in-flight client_id messages when merging server fetch into cache.
  - use-messages.ts — conversation queryFn merges cached + server on every fetch.
  - use-chat-socket.ts — WebSocket incoming messages use mergeServerMessage().
  - send onError logs to error-tracker (optional runtime log).

Files changed:
  src/lib/chat-cache.ts (new)
  src/hooks/queries/use-messages.ts
  src/hooks/use-chat-socket.ts
  src/components/chat/chat-window.tsx
  src/lib/error-tracker.ts (new, browser session log for debugging)
  src/providers/query-provider.tsx
  docs/ERROR_TRACKING.md

How to verify:
  Open a chat, send several messages quickly, wait 15+ seconds — all bubbles stay.
  DevTools: __ilmoraErrors() for runtime errors during testing.

[2026-06-02] Chat: message still disappears after ✓ (regression)
--------------------------------------------------------------------------------
Symptom:
  Same as above after first cache-merge fix; often after ~10s (poll) or when send fails.

Root causes (multiple):
  1. React Query poll/refetch replaced cache; merge in queryFn was not enough alone.
  2. send API error triggered onError → restored pre-optimistic cache (removed bubble).
  3. Production Railway used Gunicorn (WSGI only) — WebSockets/chat push did not work;
     relied on aggressive 10s HTTP polling.
  4. Gunicorn WEB_CONCURRENCY=2 + in-memory channel layer = broken WS fan-out.

Fix:
  - src/lib/pending-messages.ts — out-of-band registry; UI always merges pending sends.
  - src/hooks/use-pending-messages.ts — re-render when pending set changes.
  - chat-window.tsx — displayMessages = mergePendingMessages(); poll only if WS down.
  - use-messages.ts — no rollback on send error; mark failed + keep bubble; structuralSharing: false.
  - message-bubble.tsx — show Sending… / Failed to send.
  - backend/scripts/railway_start.sh — Daphne (config.asgi) instead of Gunicorn.

Files changed:
  (see files above + backend/scripts/railway_start.sh)

How to verify:
  Send message with WS connected (header shows "online"); bubble stays.
  If send fails, bubble turns red "Failed to send" instead of vanishing.
  Railway logs: [railway_start] Starting Daphne (HTTP + WebSocket)...

[2026-06-08] Chat: one tick on sender, message never reaches other user
--------------------------------------------------------------------------------
Symptom:
  Sender sees single checkmark; recipient never sees message. Header shows syncing.

Root causes:
  1. Single tick means "not read", not "delivered" — UI showed tick while still sending.
  2. WebSocket not connected on production (JWT / Daphne / REDIS_URL).
  3. Backend rolled back Message save when Redis broadcast failed (see backend log).

Fix:
  use-chat-socket.ts, chat-window.tsx, message-bubble.tsx, messaging/services.py
